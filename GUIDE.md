# 网文写作 Agent —— 功能总览与快速开始

一句话创意，半自动 / 全自动地写成结构完整、前后连贯的长篇网络小说。
支持命令行与本地 Web 界面两种用法。

---

## 一、它解决什么问题

长篇网文最难的不是文笔，是**连贯性**。模型上下文有限，写到第 50 章时早就看不到第 1 章——人物境界会倒退、死掉的角色会复活、伏笔会断。本系统把长篇写作拆成可控的流水线，并用三层记忆 + 双层校验兜住连贯性：

```
立项 → 分层大纲 → 单章写作 → 记忆固化 → 一致性校验 → 向量召回 → 批量续写
```

---

## 二、已实现的功能

### 1. 立项（创意 → 设定圣经）
- 输入一句话创意 + 题材，自动生成**设定圣经**（世界观、力量体系、势力、世界规则、金手指、核心矛盾、基调）
- 同时生成**初始角色库**（主角、女主等核心角色的档案）

### 2. 分层大纲
- 先生成**骨架**（每卷的弧光、目标、高潮、境界跨度）
- 再**逐卷展开**为章节细纲（每章的情节、叙事目的、爽点、章末钩子、出场角色）
- 支持只生成骨架（`--skeleton-only`）

### 3. 单章写作（完整闭环）
每写一章自动走六步闭环：

```
节奏推算 → 向量召回 → 写正文 → 抽取摘要+更新世界状态 → 一致性校验 → 落盘+索引
              │
              └─ 发现硬伤且有重写额度 → 带反馈重写
```

- 注入多层上下文：设定摘要 + 相关角色 + 本卷信息 + 近 N 章原文 + 全局记忆 + 召回的历史片段 + 节奏指引
- **关键保护**：若重写后仍有硬伤，正文保留，但不把（很可能错误的）世界状态写进记忆，等人工修正后再重写

### 4. 三层记忆系统
| 层级 | 内容 | 作用 |
|------|------|------|
| 短期 | 最近几章原文 | 相邻章节衔接自然 |
| 中期 | 滚动章节摘要 + 世界状态快照（主角境界/位置、谁生谁死、未回收伏笔） | 不读全文也掌握全局 |
| 长期 | 向量库语义召回历史片段（通义 text-embedding-v3） | 写到后期仍能"想起"早期相关场景 |

### 5. 一致性校验
- **确定性规则**：境界倒退、死人复活、角色名漂移等硬伤
- **LLM 语义校验**：时间线矛盾、金手指越界、伏笔逻辑等
- 发现硬伤自动重写；校验报告留档

### 6. 节奏引擎
- 确定性推算每章节拍（铺垫 / 蓄势 / 爆发 / 高潮 / 余韵），注入写作
- 每卷形成"铺垫→高潮→余韵喘息→再爬升"的张弛弧，避免一路平推

### 7. 批量续写（无人值守）
- 一条命令按大纲连写多章，每章走完整闭环
- 容错：单章带硬伤默认记入待办继续；写作失败则中止（可配 `--stop-on-error`）

### 8. 本地 Web 界面
- 浏览器里完成立项、生成大纲、单章/批量写作、阅读与在线编辑正文、查看世界状态与校验报告
- 写作操作弹出**进度抽屉**，通过 SSE 实时滚动每一步，结束展示用量与校验结果

## 三、快速开始

### 环境要求
- Python 3.12+
- Node.js 18+（仅用 Web 界面时需要）

### 安装

```bash
.venv/bin/pip install -e .
```

### 配置

复制 `.env.example` 为 `.env`，填入 key：

```ini
# 必填：Claude API key
ANTHROPIC_API_KEY=sk-ant-...

# 可选：走代理 / 兼容网关
# ANTHROPIC_BASE_URL=

# 向量召回需要（通义 text-embedding-v3，走 DashScope）
# 不填则自动跳过向量召回，其余功能不受影响
DASHSCOPE_API_KEY=sk-...
```

### 方式 A：命令行

```bash
# 1. 立项：一句话创意 → 设定圣经 + 角色库
novel init -i "废柴少年觉醒吞噬天赋，靠吞噬妖兽和敌人的能力逆袭" -g 玄幻 -t 吞噬万古

# 2. 生成大纲：5 卷，每卷 10 章
novel outline --book 吞噬万古 --volumes 5 --chapters 10

# 3. 写第 1 章（完整闭环）
novel write --book 吞噬万古 -c 1

# 4. 批量续写 5 章
novel run --book 吞噬万古 -n 5

# 5. 查看进度 / 导出全文
novel status --book 吞噬万古
novel export --book 吞噬万古
```

### 方式 B：Web 界面

**开发模式**（改前端即时热更新）：

```bash
# 终端 1：起后端 API（:8000）
novel serve

# 终端 2：起前端开发服务器（:5173，自动代理 /api）
cd web && npm install && npm run dev
# 浏览器打开 http://localhost:5173
```

**交付模式**（单进程，前端打包后由后端托管）：

```bash
cd web && npm run build
novel serve              # 浏览器打开 http://127.0.0.1:8000
```

## 四、命令清单

| 命令 | 作用 | 常用参数 |
|------|------|----------|
| `init` | 立项：创意 → 设定圣经 + 角色库 | `-i` 创意（必填）、`-g` 题材、`-t` 书名 |
| `outline` | 分层大纲：骨架 + 逐卷章节细纲 | `-v` 卷数、`-c` 每卷章数、`--skeleton-only` |
| `write` | 写单章（完整闭环） | `-c` 章号（0=下一章）、`-w` 字数、`--overwrite`、`--max-rewrites`、`--no-review`、`--no-vector`、`--no-consolidate` |
| `run` | 无人值守批量续写 | `-n` 章数（0=到结尾）、`-s` 起点、`-w` 字数、`--stop-on-error`、`--no-vector` |
| `reindex` | 为已写章节补建向量索引 | `--rebuild` 清空重建 |
| `serve` | 启动本地 Web 界面 | `--host`、`-p` 端口 |
| `status` | 查看设定、进度、世界状态 | `-b` 项目 slug |
| `list` | 列出所有项目 | — |
| `export` | 合并已写章节为单个 markdown | `-o` 输出路径 |
| `ping` | 测试 Claude 连通性与用量 | `-p` 测试提示词 |
| `version` | 显示版本 | — |

## 五、项目目录结构

每本书是一个自包含目录：

```
books/<书名>/
├── bible.json              # 设定圣经
├── characters.json         # 角色库
├── outline.json            # 分层大纲（卷 → 章节细纲）
├── state.json              # 世界状态快照（中期记忆）
├── summaries/chapters.json # 滚动章节摘要（中期记忆）
├── reviews.json            # 各章一致性校验报告
├── vectors.npy             # 向量库矩阵（长期记忆）
├── vectors_meta.json       # 向量库元数据
├── chapters/ch0001.md …    # 各章正文
└── 全文.md                 # export 导出的合并稿
```

## 七、模型用量

- **写作**用 Opus（质量优先），**抽取 / 校验**用 Sonnet（便宜），**向量**用通义 text-embedding-v3
- 每条命令结束打印本次 token 用量与估算成本
- 写一章（约 2500 字）大致在 $0.1 量级；带重写和校验会翻倍

---

## 八、已知限制

- 向量召回与批量续写主要做过离线验证，尚未用真实 DashScope key 大规模端到端跑通
- 大纲一次性生成固定卷数 / 章数；写到后期想加卷需手动调整 `outline.json`
- 一致性校验基于"世界状态快照 + 规则 + LLM"，没有独立的实体关系知识图谱
- 暂无自动化测试套件，质量靠人工审阅
